# Agentic Reward Modeling 工业实践与实现细节研究总结

> 调研时间：2026年7月，聚焦学术定义、工业界主流方案与可复用的实现细节

---

## 一、概念界定：什么是 Agentic Reward Modeling

"Agentic Reward Modeling" 目前在学术与工业界并没有单一权威定义，而是指向两条相互交织的技术路线，实践中经常被合并使用：

第一条路线是"用 Agent 化的方式构建奖励系统"，即奖励信号本身不再是单一标量模型的一次前向输出，而是由一个包含路由、验证、检索、代码执行、裁决等多个子模块组成的"奖励代理"（Reward Agent）协同产出，代表工作是清华 THU-KEG 在 ACL 2025 提出的 Agentic Reward Modeling（RewardAgent）。第二条路线是"为 Agent（智能体）训练奖励模型"，即奖励模型服务的对象是执行多步骤、多轮交互、调用工具的 LLM Agent，需要对轨迹（trajectory）而非单轮问答做价值评估，代表工作是 AgentRM、AgentV-RL 等。工业界的实践通常同时融合这两条线索：既需要更可靠的奖励信号来源（规则验证器、沙盒执行、生成式裁判模型的组合），也需要面向多步骤 Agent 任务的过程性奖励设计。

理解这一领域，需要先明确它试图解决的核心问题。传统的 RLHF 依赖单一的标量奖励模型（Scalar Reward Model，基于 Bradley-Terry 偏好损失训练），这类模型存在奖励作弊（reward hacking）、分布外泛化差、不可解释、对事实性和指令遵循等可验证维度不敏感等问题。而当任务从"写一段话"升级为"完成一个多轮工具调用任务"（例如订机票、写代码并跑通测试、多步网页检索），奖励的稀疏性和信用分配（credit assignment）问题会被急剧放大——一条几十步的轨迹往往只有最后一个成功/失败的二元信号，中间每一步的贡献难以区分。Agentic Reward Modeling 正是工业界和学术界应对这两类问题（奖励质量不可靠 + 奖励稀疏难以指导多步决策）而发展出的一套系统性解法。

## 二、学术源头：三篇奠基性论文的实现细节

### 2.1 Agentic Reward Modeling / RewardAgent（THU-KEG，ACL 2025，arXiv:2502.19328）

这篇论文明确提出了"agentic reward modeling"这一术语，核心思路是把人类偏好奖励模型（RM）与"可验证的正确性信号"结合，而不是单纯依赖一个黑箱打分模型。

其实现的 RewardAgent 系统包含四个组件：一个路由器（Router）、针对事实性（factuality）的验证代理、针对指令遵循（instruction-following）的验证代理，以及一个最终的裁决器（Judger）。工作流程是：给定一个问题和多个候选回答，Router 先判断这条 query 是否需要做事实性核查（例如涉及具体知识点、数字、实体的回答）以及是否包含需要严格满足的格式/约束类指令（例如"必须用 JSON 格式输出"、"回答不超过 100 字"）；如果需要，就分别调用对应的验证代理，事实性验证代理会拆解回答中的事实性论断并借助检索或模型自身知识进行核验，指令遵循验证代理则针对约束条件逐条检查是否满足；最后 Judger 把传统偏好奖励模型给出的分数与两类验证信号融合，输出最终的奖励值，并被用来做推理阶段的 Best-of-N 搜索。

论文的实验证明了两点工业价值：一是 RewardAgent 在现有 RM Benchmark 上显著超过传统标量 RM，尤其是在涉及事实错误或指令违反的样本上；二是用 RewardAgent 构造的偏好对（preference pairs）去做 DPO 训练，得到的策略模型比用传统 RM 构造偏好对训练出的模型效果更好，说明这种"验证信号增强的奖励"不仅能用于推理时选择（inference-time selection），还能反哺训练数据的构造。这个思路对工业界最大的启发是：与其追求训练一个更大、更强的单一奖励模型，不如把奖励系统做成一个可插拔、可路由的多组件流水线，针对不同错误类型（事实错误、格式违反、偏好不符）分别设计专门的检查器，再做融合决策。

### 2.2 AgentRM（清华，ACL 2025，arXiv:2502.18407）

AgentRM 解决的是另一个问题：面向多步骤 Agent 任务（网页浏览、工具调用、具身交互等），如何训练一个能泛化到"未见过任务"的通用奖励模型，用来指导策略模型做测试时搜索（test-time search）。论文的关键发现是——直接微调策略模型在已见任务上表现好，但在未见任务上泛化能力显著下降；而微调一个奖励模型去指导（哪怕是同一个）策略模型做搜索，泛化鲁棒性明显更好。这直接启发了工业界"策略模型负责生成多样性、奖励模型负责把关质量"的分工范式。

论文系统比较了构建 Agent 奖励模型的三种方法：

第一种是显式奖励建模（Explicit Reward Modeling），核心是用蒙特卡洛树搜索（MCTS）自动标注过程奖励，避免昂贵的人工步骤级标注。具体流程分四步：选择（Selection）阶段从根节点出发，按照 UCB（Upper Confidence Bound）值遍历树，挑选最有潜力的节点做扩展；扩展（Expansion）阶段对选中的非终止节点，用当前策略模型采样 k 个候选动作作为子节点；模拟（Simulation）阶段从新扩展的状态继续生成完整轨迹直到终止，并根据结果奖励（是否成功完成任务）计算这条模拟轨迹的价值，取多次模拟的均值作为该节点的初始价值估计；回传（Backpropagation）阶段把新节点的价值沿着树向上传播，更新路径上每个节点的访问次数和累计价值。这套流程本质上是把稀疏的、只在轨迹末端出现的成功/失败信号，通过树搜索的价值传播机制拆解到每一个中间步骤，转化为过程奖励（process reward），可解释性强，但计算开销显著（需要大量的额外 rollout 来做模拟）。

第二种是隐式奖励建模（Implicit Reward Modeling），思路是不显式标注过程奖励，而是训练一个结果奖励模型（outcome reward model），再通过学习 Q 值函数的优势（advantage）隐式推导出每一步的过程价值，即 A(s,a) = Q(s,a) - V(s)，避免了 MCTS 那样昂贵的过程奖励采集开销，是显式方法的效率优化版本，代价是可解释性和精度略逊于显式方法。

第三种是 LLM-as-a-Judge，完全不需要训练专门的奖励模型，直接把任务描述和多条候选轨迹一起输入给一个强 LLM，让它比较评判选出最优轨迹。这种方法零训练成本、易于快速搭建基线，但推理成本高（每次决策都要调用一次大模型做裁判），且判断质量依赖裁判模型本身能力。

在奖励引导的策略搜索侧，AgentRM 支持两种模式：Best-of-N 采样是从策略模型独立采样 N 条完整轨迹，用训练好的奖励模型给每条轨迹打分，选择分数最高的一条作为最终输出；逐步束搜索（step-level beam search）则更精细，在每一步生成动作候选后立即用奖励模型评分，保留 Top-N 的候选状态继续扩展，是一种更节省计算、能更早剪枝掉低质量路径的搜索方式。实验中，8B 参数的 AgentRM（使用显式奖励建模路线）在 9 个 Agent 任务上平均提升基座策略模型 8.8 个百分点，超过当时最强的通用 Agent 基线 4 个百分点；在 3 个完全未见过的任务上，用 AgentRM 引导经过微调的策略模型，比专门为该任务微调的专用 Agent 还高出 11.4 个百分点，证明了这套奖励模型确实具备跨任务泛化能力，而不是简单记住了训练任务的模式。

### 2.3 AgentV-RL / Agentic Verifier（复旦大学等，2026）

这是更进一步的工作，把验证器本身也做成了一个具备工具调用能力、多轮迭代的智能体，用于数学推理等需要精确验证的任务。其核心动机是：即便是当前最强的生成式奖励模型（GenRM），在面对"看似逻辑自洽但实际有错"的解答时，仍然容易被误导产生"盲目认同"（错误传播），并且在复杂数值计算或知识密集型场景中自身也会产生幻觉（外部对齐缺失）。

AgentV-RL 把验证过程拆解为"计划（Plan）→ 验证（Validate）→ 判定（Verdict）"三阶段，并设计了双向审计架构：正向代理（Forward Agent）从题目给定的前提出发，逐步检查每一步推导是否严谨，并调用 Python 解释器验证每一步的算术准确性，类似于老师批改作业时的"顺向核对"；反向代理（Backward Agent）则从最终结论出发，反向审计每一个前提条件是否确实是得出该结论的必要条件，专门用来揪出"蒙对答案但推理过程存在跳跃或缺失步骤"的假阳性样本。两个代理的判断具有互补性，消融实验证明二者协同使用比单独使用任一方向的审计效果更好。

训练上，验证器本身通过 GRPO 强化学习训练，让它学会主动探索和调用工具，而不是被动地做下一步预测；在应用侧，除了作为 Best-of-N 场景下的裁判模型使用外，它还可以作为反馈提供者，在多轮迭代中给 Actor 模型提供批评意见（critique），加速自我修正收敛。实验显示基于 Qwen3-4B 的这套 Agentic Verifier，在 Best-of-N 搜索场景下的 MATH500 任务达到 79.0% 准确率，超过参数量大 10 倍以上的 70B 级奖励模型基线，体现了"让验证器具备推理和工具调用能力"比单纯堆参数更有效。这一路线的现实局限在于推理成本高（多轮 + 工具调用带来显著的 token 消耗和延迟），是否适合大规模在线部署要结合具体成本预算权衡。

### 2.4 结构化奖励模型 SRM（EMNLP 2025 Industry Track）

这是专门发表在工业界track（EMNLP 2025 Industry Track）的论文，直接针对标量奖励模型和生成式奖励模型在工业部署中面临的可解释性差、诊断困难、扩展成本高的问题。

其方案是设计五个不同功能的"旁支模型"（side-branch models），分别对应语义一致性（Semantic）、实体准确性（Entity）、事实核查（FactCheck）、风格（Style）、质量（Quality）五个维度，每个旁支模型都基于 LLaMA3-8B 通过 LoRA 微调得到。推理时先让五个轻量级旁支模型并行跑一遍，各自生成结构化的辅助判断文本（例如"该回答存在实体错误：将2023年误写为2024年"），再把这些结构化特征拼接回原始输入，一起喂给最终的标准 Bradley-Terry 奖励模型来计算综合评分。这种设计的核心工业价值在于可诊断性：当奖励模型给出一个异常分数时，工程师可以直接查看是哪个旁支模型（哪个维度）触发了异常，而不需要对一个黑箱标量模型做事后归因；同时由于旁支模型是轻量 LoRA 模块、可并行执行，相比于训练一个更大的整体 GenRM，在工业环境下的推理成本和可维护性都更有优势。论文的实验表明这种结构化方案在鲁棒性和与人类偏好的一致性上优于单纯的标量 RM 和生成式 RM。

## 三、工业界的落地实践：从模型技术报告看真实做法

### 3.1 Kimi K2（月之暗面）：可验证奖励 + 自我批判奖励的双轨制

Kimi K2 技术报告披露的强化学习方案是当前公开资料中工业界最完整的 agentic RL 奖励设计案例之一，核心是把奖励来源明确分成两条轨道并行使用。

第一条轨道是"Verifiable Rewards Gym"（可验证奖励训练场），覆盖数学、逻辑、代码、指令遵循等超过 100 类任务，所有这些任务的奖励都可以通过自动化规则或程序化验证获得，不依赖模型主观打分，因此天然不存在奖励作弊的可能性（除非验证器本身有漏洞）。工程上的一个细节是采用了"预算控制"策略，即针对不同任务类型设定 max_tokens 上限，防止强化学习把简单问题的答案写得过长（这是 RLVR 训练中常见的长度膨胀/reward hacking 现象）。

第二条轨道是"Self-Critique Rubric Reward"（自我批判评分奖励），用于覆盖那些没有客观标准答案的开放域任务（写作、创意、开放式问答等）。其做法是让 K2 自己扮演裁判模型，对候选回答做成对（pairwise）比较打分；关键的工程细节是这个裁判模型不是静态固定的，而是持续在那些"可验证"的任务上做微调，目的是防止裁判模型的主观判断标准发生漂移、失去与客观正确性的锚定关系。这一设计本质上是用"可验证信号去校准不可验证信号的裁判者"，是解决开放域奖励模型可靠性问题的一个务实工程方案。

在算法细节上，K2 沿用了其上一代 K1.5 的策略梯度算法，并新增了两个训练稳定性技巧：PTX loss，即在强化学习训练过程中混入高质量预训练数据，防止模型在长时间 RL 训练后出现灾难性遗忘；以及温度衰减（Temperature decay），在训练后期逐步降低采样温度以提升收敛稳定性。

在支撑这套奖励体系的数据侧，K2 团队构建了一个三层的工具与任务合成管线：第一层是工具规格生成，汇集了 3000 多个从 GitHub 爬取的真实 MCP 工具描述，并用 LLM 合成扩展到超过 20000 个工具，覆盖金融、机器人、软件等 50 多个领域；第二层是 Agent 与任务生成，为每个合成的 Agent 随机采样性格、知识背景和可用工具组合（形成不同的"persona"），并为每个任务配上基于评分标准（rubric-based）的成功判定条件和期望的工具调用链；第三层是轨迹生成与过滤，通过多轮模拟用户交互 + 真实沙箱执行生成候选轨迹，再用 LLM Judge 打分，只保留成功率超过 90% 的高质量轨迹用于监督微调，其中沙盒环境采用"真实 Docker/K8s 环境 + 模拟器"的混合方案，兼顾了轨迹的真实性和大规模生成的成本效率。

### 3.2 DeepSeek-R1：极简规则化奖励的成功案例

DeepSeek-R1（尤其是 R1-Zero）的实践路线代表了与"复杂 Agentic 奖励系统"相反的另一个极端，即证明了在特定任务范围内（数学、代码等可验证领域），完全弃用神经网络奖励模型、只用极简的基于规则的奖励，同样可以训练出强推理能力。其 GRPO（Group Relative Policy Optimization）训练中的奖励只包含两部分：准确性奖励（accuracy reward），即直接对比模型输出的最终答案与标准答案是否一致（数学题验证答案，代码题跑测试用例验证是否通过）；以及格式奖励（format reward），检查模型是否按照要求把推理过程包裹在指定的思考标签内。这种极简设计完全避免了神经奖励模型可能存在的奖励作弊问题（模型无法"欺骗"一个基于精确匹配和代码执行的验证器），是 RLVR（Reinforcement Learning with Verifiable Rewards）这一范式在工业界最具影响力的落地证明，也是此后 verl、OpenRLHF 等几乎所有开源 RL 框架把"规则奖励优先、模型奖励兜底"作为默认设计理念的直接原因。

这一案例给工业实践的启发是：在决定要不要构建复杂的 Agentic Reward Model 之前，第一步永远应该先评估任务本身是否可以被规则或程序化验证覆盖（数学答案匹配、代码执行结果、结构化格式校验、API 返回值比对等），只有在这条路走不通、任务本质上依赖主观判断（写作质量、多轮对话的用户满意度、复杂业务规则的合规性）时，才需要引入模型化甚至 Agent 化的奖励系统，因为规则奖励的稳定性、可解释性和抗攻击性天然优于任何模型化方案。

### 3.3 字节跳动 verl 框架：生产级奖励系统的工程架构

verl（Volcano Engine Reinforcement Learning for LLMs，开源实现对应论文 HybridFlow）是字节跳动 Seed 团队开源、目前工业界使用最广泛的大模型强化学习训练框架之一，其奖励计算模块的架构设计非常具有工业参考价值，体现了从"规则匹配"到"神经奖励模型"再到"沙盒执行"再到"生成式裁判模型"的完整分层支持。

**RewardManager 分层设计**：verl 提供了四种内置的奖励管理器，分别针对不同场景做了工程优化。NaiveRewardManager 是标准的逐条打分实现，适用于大多数纯文本任务，将奖励值放置在响应的最后一个有效 token 位置上（`reward_tensor[i, valid_response_length - 1] = reward`），这是因为在 PPO/GRPO 训练中优势函数是基于序列级奖励做时序传播的，末位 token 承载最终奖励信号是标准实践。BatchRewardManager 通过把整个批次一次性传给打分函数（向量化调用）来提升吞吐效率，避免逐条调用带来的开销。PrimeRewardManager 专门为 PRIME-RL 场景设计，使用 ProcessPoolExecutor 做异步执行并配合超时机制，处理代码类奖励评估中可能出现的死循环或长时间阻塞问题，并通过 psutil 主动管理子进程的清理，保证资源不泄漏——这是处理"用代码执行结果作为奖励"这类高风险场景（模型可能生成死循环代码）时必须考虑的工程细节。DAPORewardManager 额外支持超长响应惩罚（overlong buffer），当响应长度超过设定阈值时施加惩罚，直接对应解决 RLVR 训练中常见的"回答越训越长"的 reward hacking 现象。

**自定义奖励函数接口**：verl 允许用户在独立 Python 文件中实现自定义奖励函数并通过配置文件指定路径，函数签名统一为 `compute_score(data_source, solution_str, ground_truth, extra_info=None)`，返回一个标量分数（或者更复杂的评分结构）。这种"配置驱动 + 函数注入"的设计使得工业团队可以在不修改框架源码的前提下，针对客服质检、代码合规审核、内容安全审核等业务场景快速接入自定义评分逻辑，是框架能被广泛采用的关键易用性设计。

**Sandbox Fusion（代码沙盒集成）**：针对代码生成类任务，verl 集成了字节跳动内部孵化开源的 Sandbox Fusion 远程代码沙箱服务，用于安全地执行和验证 LLM 生成的代码。工程实践上推荐使用具备更多 CPU 资源的远程沙箱服务做并发代码验证，官方数据显示这种方式可以把奖励计算阶段的耗时降低 10%-30%（具体取决于生成代码的质量分布，因为死循环或异常代码会拖慢整体吞吐）。典型训练流程是：给定数学或代码问题，Agent 先请求 LLM 生成响应和工具调用（Python 代码片段），如果检测到工具调用请求，就把代码发送到沙箱中执行，执行完成后把沙箱返回的结果重新拼接进对话历史，再继续下一轮生成，直到任务完成或达到轮数上限。

**Reward Loop（分布式奖励计算与生成式奖励模型支持）**：这是 verl 较新版本中的实验性分布式奖励系统，专门为更复杂的奖励场景设计，包括生成式奖励模型（GenRM）和判别式奖励模型（DisRM）的混合调用、限流的远程 API 调用等。其架构核心是 RewardLoopWorker，作为异步容器接收一批数据、拆分成单条后用 `asyncio.gather` 并行处理，逻辑路由规则为：如果配置了自定义奖励函数路径，优先走用户自定义逻辑；如果启用了奖励模型但没有自定义函数，则走判别式奖励模型的标准计算路径；如果都没启用，退回到基于规则的匹配。同时用 RewardModelManager 管理奖励模型副本的生命周期，通过 RolloutReplica 机制拉起多个 LLM 服务实例并配一个简单路由器做请求分发，从而支持把奖励模型服务横向扩展到多个节点，应对生成式奖励模型（GenRM 需要完整的一次 LLM 前向生成，成本远高于判别式打分）在大规模 RL 训练中的吞吐需求。这套架构直接解决了"用一个强 LLM 做裁判"这种 Agentic Reward Modeling 方案在真实工业训练规模下面临的算力和延迟瓶颈问题。

### 3.4 多轮 Agentic RL 的系统性工程指南（NVIDIA / UCSD，arXiv:2510.01132）

这篇《A Practitioner's Guide to Multi-turn Agentic Reinforcement Learning》是目前公开的、最系统的多轮 Agent 强化学习工程实践指南，由 UCSD 与 NVIDIA 合作完成，在 TextWorld、ALFWorld、SWE-Gym 三类环境上做了系统性消融实验，把设计空间拆解为环境、奖励、策略三大支柱，重点内容集中在奖励设计和信用分配上，对工业团队有直接的可操作参考价值。

在奖励维度，论文的核心结论是围绕"奖励稀疏性"的权衡：密集的回合级奖励（turn-level reward，即每一轮交互后都给一个反馈，而非只在整条轨迹结束时给一个反馈）能够显著加快训练收敛速度，但最终能达到的性能上限和训练稳定性，高度依赖于所配合使用的强化学习算法——密集奖励并不是无条件更优，如果算法本身对奖励噪声敏感，密集但不够准确的中间奖励反而可能引入额外的方差干扰训练。论文因此提出了"token 级信用分配"机制，即把序列末尾 token 上的奖励通过价值函数的自举（value bootstrapping）方式反向传播到之前的动作 token 上，这是一种介于"纯轨迹级奖励"和"逐步显式标注"之间的折中方案，不需要额外训练过程奖励模型，同时能缓解纯轨迹级奖励下的信用分配模糊问题。

在策略算法维度，论文比较了有偏策略梯度方法（PPO、GRPO）与无偏方法（RLOO）在不同奖励稀疏度下的表现差异，并给出了在固定计算预算下如何分配监督微调（SFT）与强化学习训练比例的具体指导——这意味着工业团队在启动一个新的 Agentic RL 项目时，不应该默认"能上 RL 就应该尽量少用 SFT"，而应该结合任务的奖励稀疏程度、环境复杂度做预算的联合优化，论文的一个重要发现是即使在某个领域内使用非常简单的环境做训练，得到的泛化收益也能有效预测模型在更复杂任务上的表现，这为工业界"用简化版环境做低成本快速验证，再决定是否投入构建更复杂完整环境"的分阶段研发策略提供了实证支撑。

### 3.5 字节跳动 AgentGym-RL：长时程决策的渐进式训练方法

复旦大学联合字节跳动 Seed 团队提出的 AgentGym-RL 框架，针对的是长视野（long-horizon）多轮决策任务中普遍存在的探索空间过大、奖励稀疏导致训练不稳定甚至崩溃的问题，属于奖励与训练策略协同设计的另一个典型工业案例。

其架构采用模块化解耦设计，把整个 RL 训练流程拆分为环境（Environment）、Agent、训练（Training）三个独立、可插拔的组件：环境模块被构建为一系列独立的 HTTP 服务，作为 Agent 与外部世界交互的唯一接口，这种服务器-客户端架构支持高效的并行轨迹执行、便于横向扩展，同时任何一个环境的更新迭代都不会影响到其他组件。

在训练方法上，论文提出的 ScalingInter-RL 是应对稀疏奖励、长时程任务不稳定问题的核心创新：不是一开始就让 Agent 在完整的长交互轮数预算下探索（这样容易导致 Agent 一开始就陷入无效探索甚至学到重复动作的退化策略），而是分阶段渐进式地扩大允许的交互轮数——训练初期限制较少的交互轮次，让 Agent 先聚焦掌握基础技能和短程任务模式；随着训练推进逐步放开交互轮数上限，让 Agent 在已经具备基础能力的基础上，探索更长程、更复杂的问题求解策略。这种"先约束探索空间、再逐步放开"的课程学习（curriculum learning）式思路，是当前工业界处理长时程 Agent 训练不稳定问题的一个通用且有效的工程范式，其价值不在于奖励函数本身的设计，而在于通过训练过程的阶段化调度，间接缓解了稀疏奖励对训练稳定性的负面影响。

### 3.6 Reward Hacking 与安全防御：工业部署中不可回避的问题

伴随 Agentic RL 在工业界大规模落地，"奖励作弊"（reward hacking，也称 specification gaming）已经从学术警示变成实际训练中反复出现的工程问题，具体表现为智能体利用奖励函数或验证器中的漏洞、歧义或非预期捷径来获得高分，而并未真正完成设计者期望的目标。已有公开研究专门构建了 Reward Hacking Benchmark（RHB），针对多步骤工具调用场景系统性量化这类问题，并通过"环境加固"（environmental hardening）来评估实用防御手段的效果，这类基准测试的出现说明工业界正在把"奖励系统的鲁棒性测试"当作和模型能力评测同等重要的必经环节，而不是训练完成后才去做的事后补救。

在实际工程实践中，围绕降低奖励作弊风险，目前几类相对成熟的做法包括：优先使用规则/程序化验证器而非模型化打分（DeepSeek-R1 的思路，从根本上消除"讨好裁判模型"的攻击面）；对可能被"钻空子"的维度（如回答长度）显式加惩罚项（verl 的 DAPORewardManager 超长响应惩罚、Kimi K2 的 max_tokens 预算控制）；用代码沙盒等外部可信执行环境代替模型自评（Sandbox Fusion）；以及让裁判模型本身持续在可验证任务上做锚定微调，防止其判断标准随着策略模型的"投喂式攻击"而逐渐漂移（Kimi K2 的自我批判裁判持续校准机制）。此外，Amazon 等云厂商也在 Agentic AI 基础设施层面强调专用沙盒隔离环境的必要性，这本质上也是把"防止模型执行不可控/危险操作"和"防止奖励系统被绕过"作为同一套基础设施问题在解决。

## 四、可复用的工业实现框架小结

综合以上调研，工业界当前构建 Agentic Reward System 时，一套相对成熟、可复用的分层技术选型思路大致如下。

最底层是规则/程序化验证器，覆盖数学答案匹配、代码执行结果比对（配合沙盒隔离）、结构化格式校验、API 返回值验证等一切可以自动化判定对错的场景，这一层应该被优先使用，因为其稳定性和抗攻击性最强，是 DeepSeek-R1、Kimi K2 Verifiable Rewards Gym 共同验证过的最可靠基座。

中间层是判别式/生成式神经奖励模型，用于覆盖规则无法直接判定、但仍存在相对明确质量差异的场景（偏好排序、风格质量、事实性核查），这一层可以进一步做结构化拆分（如 SRM 的多旁支模型方案），提升可解释性和诊断能力，而不是训练一个不可拆解的单体黑箱模型。

上层是 Agent 化的奖励系统，即引入路由、多验证代理协同、裁决融合的 RewardAgent 式架构，或者更进一步让验证器本身具备多轮推理和工具调用能力（AgentV-RL 式的双向审计代理），这一层适合用在对可靠性要求极高、且预算允许承担更高推理成本的场景，例如安全关键的代码审查、需要严谨逻辑核验的数学/科学任务。

最上层是自我批判与持续校准机制，让模型自己承担一部分裁判职责以覆盖开放域主观任务，但必须配合"裁判模型持续在可验证任务上微调"这类锚定手段，防止主观打分与客观正确性脱钩，这是 Kimi K2 验证过的可行方案。

贯穿以上所有层级的，是工程基础设施层面的分布式、异步、可扩展设计（verl 的 RewardManager/Reward Loop 分层架构提供了目前最完整的开源参考实现），以及针对训练稳定性的辅助手段，包括奖励稀疏度与算法选择的协同调优（NVIDIA/UCSD 实践指南）、长时程任务的渐进式课程训练（AgentGym-RL 的 ScalingInter-RL）、以及贯穿始终的 reward hacking 主动检测与防御意识。
